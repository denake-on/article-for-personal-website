## 一、基本信息
### 背景
本项目使用单张的a800显卡，80G显存，基于veRL框架中的GRPO算法，对qwen2.5-3B-instruct模型进行了全参数微调，实现了完整的post-training流程，包括SFT冷启动、reward函数设计以及GRPO强化学习。
### 成果
- 在训练数据集medqa上
	- baseline模型准确率仅30%，通过SFT冷启动之后，模型学会了按照规定的格式输出，准确率跃升至38%。通过6个epoch的grpo算法训练，模型在该数据集上的准确率达到了48%，跃升至10个百分点。
- 在分布外数据集 MMLU-Health上
	- GRPO 模型准确率较 SFT提升 8 个百分点，证实 RL 学到的推理能力更具鲁棒性。


## 二、关于veRL与强化学习基础
### 1. GRPO算法概览
**GRPO (Group Relative Policy Optimization)** 是由DeepSeek团队在论文[DeepSeekMath: Pushing the Limits of Mathematical Reasoning in Open Language Models](https://arxiv.org/pdf/2402.03300)中提出的强化学习算法。

GRPO的核心思想是：**消除对价值函数（Critic）的依赖**，通过组内相对奖励来计算优势函数，从而简化训练过程并减少计算开销。

**传统ppo算法训练**
```
                    传统 PPO for LLM
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  输入 Prompt                                                 │
│      │                                                       │
│      ▼                                                       │
│  ┌─────────────┐     生成 G 个候选回复                       │
│  │ Actor (LLM) │ ───────────────┐                           │
│  └─────────────┘                │                           │
│      │                          ▼                           │
│      │                   ┌──────────┐                        │
│      │                   │   RM     │ 计算奖励               │
│      │                   └──────────┘                        │
│      │                          │                           │
│      ▼                          ▼                           │
│  ┌─────────────┐         ┌──────────┐                        │
│  │  Critic     │◄────────│   训练   │ 需要价值估计!          │
│  │ (Value Net) │         └──────────┘                        │
│  └─────────────┘                                              │
│      │                                                       │
│      └──→ 更新 Actor                                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```
缺点在于critic网络需要大量计算资源，并且器降至估计在语言生成任务中往往不够准确。

**GRPO算法训练**
对于语言生成任务，可以通过组内相对比较来估计优势，而不需要显式的价值函数。
```
                    GRPO for LLM
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  输入 Prompt                                                 │
│      │                                                       │
│      ▼                                                       │
│  ┌─────────────┐     生成 G 个候选回复 (组采样)              │
│  │ Actor (LLM) │ ───────────────┐                           │
│  └─────────────┘                │                           │
│                                  ▼                           │
│                           ┌──────────┐                        │
│                           │   RM     │ 计算每个回复的奖励     │
│                           └──────────┘                        │
│                                  │                           │
│                                  ▼                           │
│                        ┌──────────────────┐                   │
│                        │  组内优势估计    │                   │
│                        │  (无需Critic!)   │                   │
│                        └──────────────────┘                   │
│                                  │                           │
│                                  ▼                           │
│                           ┌──────────┐                        │
│                           │  训练    │ 只更新Actor            │
│                           └──────────┘                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.GRPO算法具体原理
#### 2.2.1 组采样
$$
(y_1, y_2, \ldots, y_G) \sim \pi_\theta(\cdot|x)
$$
1.  **输入一个Prompt ($x$)**: 假设你给大语言模型一个问题，比如 "解释量子纠缠"。这就是你的prompt $x$。
2.  **模型生成多个回复**: 你的大语言模型（也就是Actor网络 $\pi_\theta$）会根据这个prompt $x$ 生成 $G$ 个**不同的**输出（回复）。
    *   这些输出可以是 $y_1, y_2, \ldots, y_G$。
    *   例如，模型可能会生成：
        *   $y_1$: "量子纠缠是两个粒子之间的一种特殊联系..."
        *   $y_2$: "当两个量子系统纠缠在一起时，它们的状态是相互关联的..."
        *   $y_3$: "爱因斯坦称之为‘鬼魅般的超距作用’..."
        *   ...直到 $y_G$。
    *   **关键点**: 这些回复都是由**同一个模型**（当前训练中的Actor $\pi_\theta$）在**同一个prompt** $x$ 下生成的。它们之所以不同，是因为生成过程中通常会引入一些随机性（例如，通过温度采样、top-k采样等）。
3.  **形成一个“组”**: 这 $G$ 个回复 $(y_1, y_2, \ldots, y_G)$ 就构成了一个“组”。这个组的特点是：它们都试图回答同一个问题，因此它们之间是可比较的。
#### 2.2.2 组内优势估计
对于组内的每个输出 $y_g$，其优势函数计算为：

$$
\boxed{A_g = \frac{R(y_g|x) - \text{baseline}}{\sigma(R)}}
$$

其中：

- **$R(y_g|x)$**: 第 $g$ 个输出从奖励模型获得的奖励
- **$\text{baseline}$**: 组内奖励的基线（通常使用组内平均奖励）
- **$\sigma(R)$**: 组内奖励的标准差（用于归一化）

具体展开：

$$
\text{baseline} = \frac{1}{G} \sum_{i=1}^{G} R(y_i|x)
$$

$$
\sigma(R) = \sqrt{\frac{1}{G} \sum_{i=1}^{G} (R(y_i|x) - \text{baseline})^2}
$$

因此优势函数的完整表达式为：

$$
A_g = \frac{R(y_g|x) - \frac{1}{G}\sum_{i=1}^{G} R(y_i|x)}{\sqrt{\frac{1}{G}\sum_{i=1}^{G} \left(R(y_i|x) - \frac{1}{G}\sum_{j=1}^{G} R(y_j|x)\right)^2}}
$$
最后我们分析$A_g$:
*   如果 $R(y_g|x)$ **高于** $\text{baseline}$，那么 $A_g$ 就是**正值**，表示这个回复比组内平均水平要好。模型应该更多地生成这样的回复。
*   如果 $R(y_g|x)$ **低于** $\text{baseline}$，那么 $A_g$ 就是**负值**，表示这个回复比组内平均水平要差。模型应该减少生成这样的回复。
*   如果 $R(y_g|x)$ **等于** $\text{baseline}$，那么 $A_g$ 就是**零**，表示这个回复处于组内平均水平。

#### 2.2.3 目标函数
GRPO的目标函数类似于PPO，但使用组内估计的优势：
$$
L^{GRPO}(\theta) = \mathbb{E}_{x \sim \mathcal{D}} \left[ \mathbb{E}_{(y_1,\ldots,y_G) \sim \pi_\theta(\cdot|x)} \left[ \sum_{g=1}^{G} \frac{\pi_\theta(y_g|x)}{\pi_{\theta_{ref}}(y_g|x)} A_g \right] \right]
$$
为了稳定性，通常使用CLIP版本：
$$
\boxed{L^{GRPO}_{CLIP}(\theta) = \mathbb{E}\left[ \sum_{g=1}^{G} \min\left(r_g A_g, \text{clip}(r_g, 1-\epsilon, 1+\epsilon) A_g\right) \right]}
$$
1.  **$\mathbb{E}[\ldots]$**:
    *   这表示对所有可能的prompt $x$ 和在每个prompt下生成的所有组 $(y_1, \ldots, y_G)$ 进行期望。简单来说，就是对所有训练数据和生成样本的平均。

2.  **$\sum_{g=1}^{G} \ldots$**:
    *   这表示对组内的每一个样本 $y_g$ 进行求和。GRPO的“组采样”特性在这里体现，它同时考虑了同一个prompt下生成的 $G$ 个不同回复。

3.  **$r_g = \frac{\pi_\theta(y_g|x)}{\pi_{\theta_{ref}}(y_g|x)}$**:
    *   这是**概率比率 (Probability Ratio)**。
    *   $\pi_\theta(y_g|x)$ 是当前策略 $\theta$ 生成 $y_g$ 的概率。
    *   $\pi_{\theta_{ref}}(y_g|x)$ 是参考策略 $\theta_{ref}$ 生成 $y_g$ 的概率。
    *   **作用**: 这个比率衡量了当前策略相对于参考策略，生成某个特定输出 $y_g$ 的倾向性变化。
        *   如果 $r_g > 1$，说明当前策略更倾向于生成 $y_g$。
        *   如果 $r_g < 1$，说明当前策略更不倾向于生成 $y_g$。
        *   如果 $r_g = 1$，说明当前策略和参考策略生成 $y_g$ 的概率相同。
    *   **为什么是 $\pi_{\theta_{ref}}$ 而不是 $\pi_{\theta_{old}}$？** 这是GRPO的一个重要区别。在PPO中，通常使用上一个迭代的策略 $\pi_{\theta_{old}}$。GRPO使用一个固定的或缓慢更新的**参考策略** $\pi_{\theta_{ref}}$。这有助于防止“模式崩塌”（mode collapse），即策略过度集中于少数高奖励的输出，导致生成多样性下降。参考策略提供了一个稳定的基准。

4.  **$A_g$**:
    *   这是**组内优势函数 (Group-wise Advantage Function)**。
    *   根据你提供的上下文，它的计算方式是：$A_g = \frac{R(y_g|x) - \text{baseline}}{\sigma(R)}$。
    *   **作用**: $A_g$ 衡量了输出 $y_g$ 相对于同组其他输出的“好坏程度”。
        *   如果 $A_g > 0$，说明 $y_g$ 的奖励高于组内平均水平，是一个“好”的输出。
        *   如果 $A_g < 0$，说明 $y_g$ 的奖励低于组内平均水平，是一个“坏”的输出。
        *   如果 $A_g = 0$，说明 $y_g$ 的奖励与组内平均水平相当。
    *   **关键点**: GRPO通过这种方式**移除了对Critic网络的依赖**。它不再需要一个单独的网络来估计状态价值 $V(s)$ 或状态-动作价值 $Q(s,a)$，而是通过组内样本的相对比较来估计优势。

5.  **$\min\left(r_g A_g, \text{clip}(r_g, 1-\epsilon, 1+\epsilon) A_g\right)$**:
    *   这是PPO中经典的**裁剪 (Clipping)** 机制，GRPO也沿用了它。
    *   **$\text{clip}(r_g, 1-\epsilon, 1+\epsilon)$**: 这个函数将概率比率 $r_g$ 限制在一个区间 $[1-\epsilon, 1+\epsilon]$ 内。$\epsilon$ 是一个小的超参数（例如0.1或0.2）。
        *   如果 $r_g$ 在这个区间内，那么 $\text{clip}(r_g, \ldots)$ 就等于 $r_g$。
        *   如果 $r_g > 1+\epsilon$，它会被裁剪到 $1+\epsilon$。
        *   如果 $r_g < 1-\epsilon$，它会被裁剪到 $1-\epsilon$。
    *   **作用**: 裁剪机制是为了**限制策略更新的幅度**。
        *   当 $A_g > 0$ (好样本) 时：我们希望增加生成 $y_g$ 的概率。如果 $r_g$ 变得非常大（策略变化太大），裁剪会限制这种增长，防止策略过度优化导致不稳定。
        *   当 $A_g < 0$ (坏样本) 时：我们希望减少生成 $y_g$ 的概率。如果 $r_g$ 变得非常小（策略变化太大），裁剪会限制这种下降，防止策略过度惩罚导致不稳定。
    *   **$\min(\ldots)$ 的作用**:
        *   如果 $A_g > 0$ (好样本): 我们希望 $r_g$ 越大越好，但不能超过 $1+\epsilon$。$\min$ 函数会选择 $r_g A_g$ 和 $(1+\epsilon) A_g$ 中较小的一个。这意味着，即使 $r_g$ 变得非常大，我们获得的收益也不会超过 $(1+\epsilon) A_g$，从而限制了策略更新的上限。
        *   如果 $A_g < 0$ (坏样本): 我们希望 $r_g$ 越小越好，但不能低于 $1-\epsilon$。$\min$ 函数会选择 $r_g A_g$ 和 $(1-\epsilon) A_g$ 中较小的一个。由于 $A_g$ 是负数，这意味着我们希望 $r_g$ 尽可能小，但不能让损失变得过大（即 $r_g A_g$ 不能比 $(1-\epsilon) A_g$ 更负）。这限制了策略更新的下限。

##### 另外需要添加KL散度防止策略更新幅度过大
   $$
   D_{KL}(\pi_\theta || \pi_{ref}) = \mathbb{E}_{x,y}\left[\log \frac{\pi_\theta(y|x)}{\pi_{ref}(y|x)}\right]
   $$
   防止策略偏离参考策略太远

#### 2.2.4 veRL中的GRPO算法实现
##### 2.4.1 核心代码结构
verl框架中GRPO的核心实现位于：
- `verl/trainer/config/algorithm.py` - 算法配置
- `verl/trainer/ppo/core_algos.py` - 核心算法实现
- `verl/trainer/ppo/ray_trainer.py` - 训练循环
##### 2.4.2 优势估计器实现
在 `core_algos.py` 中，GRPO通过 `compute_grpo_outcome_advantage` 函数实现：

```python
@register_adv_est(AdvantageEstimator.GRPO)
def compute_grpo_outcome_advantage(
    token_level_rewards: torch.Tensor,  # (batch_size, response_length)
    response_mask: torch.Tensor,         # (batch_size, response_length)
    index: np.ndarray,                   # group index数组
    epsilon: float = 1e-6,
    norm_adv_by_std_in_grpo: bool = True,
    config: Optional[AlgoConfig] = None,
) -> tuple[torch.Tensor, torch.Tensor]:
    """
    计算GRPO优势函数
    """
    # 1. 对每个sequence的token-level奖励求和，得到sequence-level分数
    scores = token_level_rewards.sum(dim=-1)  # (batch_size,)

    # 2. 按group index分组，计算组内均值和标准差
    id2score = defaultdict(list)
    id2mean = {}
    id2std = {}

    for i in range(batch_size):
        id2score[index[i]].append(scores[i])

    for idx in id2score:
        if len(id2score[idx]) == 1:
            # 只有一个样本时，标准差设为1避免除零
            id2mean[idx] = torch.tensor(0.0)
            id2std[idx] = torch.tensor(1.0)
        elif len(id2score[idx]) > 1:
            scores_tensor = torch.stack(id2score[idx])
            id2mean[idx] = torch.mean(scores_tensor)
            id2std[idx] = torch.std(scores_tensor)

    # 3. 计算归一化优势
    for i in range(batch_size):
        if norm_adv_by_std_in_grpo:
            # 原始GRPO: a_i = (r_i - mu) / sigma
            scores[i] = (scores[i] - id2mean[index[i]]) / (id2std[index[i]] + epsilon)
        else:
            # DrGRPO: a_i = r_i - mu
            scores[i] = scores[i] - id2mean[index[i]]

    # 4. 将标量优势广播到token维度
    scores = scores.unsqueeze(-1) * response_mask  # (batch_size, response_length)

    return scores, scores
```

### 3.5 RL阶段
#### 3.5.1 reward设计
**第一阶段**：
reward主要通过两个方面给出，分别是format reward和correctness reward。
- format
	- 在SFT阶段得到的模型回答中采样了回答的长度范围，因此确定下来回答的整体长度需要在规定范围内（100-2000字符），过长或者过短都不符合要求。
	- 必须按照推理过程+Final answer的形式给出回答
- correctness
	- 模型给出的结果需要和ground truth一致。

奖励机制：  
- 符合格式 + 正确答案：+1.0  
- 符合格式 + 错误答案：0.0

### 四、遇到的问题
#### OOM 和 Socket 关闭错误（显存未满但系统内存爆满）
##### 问题描述

 第一阶段：系统内存 OOM

##### 原因分析
通过 `nvidia-smi` 观察：
```
GPU 显存占用：1402MiB / 81920MiB (1.7%)  ← 显存几乎空闲
GPU 利用率：高（56%-91%）                    ← GPU 在工作
```

**结论**：问题不是 GPU 显存不足，而是**系统内存（RAM）不足**

##### 错误配置
原本的脚本启用了 CPU Offload：
```bash
actor_rollout_ref.actor.fsdp_config.param_offload=True          # ❌ 参数 offload 到 RAM
actor_rollout_ref.actor.fsdp_config.optimizer_offload=True      # ❌ 优化器 offload 到 RAM
```

##### 内存占用分析
启用 CPU Offload 后的内存分布：
```
Actor 模型参数 (fp32):    3B × 4 bytes = 12 GB → offload 到 RAM
优化器状态 (Adam):        2 × 12 GB = 24 GB → offload 到 RAM
Reference 模型:           ~2 GB → 部分在 RAM
Reward Workers:           ~4 GB → RAM
vLLM Rollout:             ~2 GB → RAM
其他 Ray overhead:        ~10 GB → RAM
────────────────────────────────────────────
总计系统内存占用：         ~54 GB（仅模型相关）
加上 FSDP 分布式开销：     ~98 GB（实际观察 PID 4376）
```

系统内存只有 120 GB，当训练需要同时加载 Actor + Rollout + Reference 时，内存需求超过 116 GB，触发 OOM killer 杀死进程。

##### 解决方案
关闭 CPU Offload

**修改前（错误配置）**：
```bash
actor_rollout_ref.actor.fsdp_config.param_offload=True
actor_rollout_ref.actor.fsdp_config.optimizer_offload=True
```

**修改后（正确配置）**：
```bash
actor_rollout_ref.actor.fsdp_config.param_offload=False      # ✅ 保留在 GPU 显存
actor_rollout_ref.actor.fsdp_config.optimizer_offload=False  # ✅ 保留在 GPU 显存
```

##### 解释：
FSDP 是 PyTorch 提供的一种分布式训练策略，旨在解决大规模模型训练时显存不足的问题。它通过将模型参数、梯度和优化器状态在不同的GPU之间进行分片（sharding），从而大大减少每个GPU的显存占用。

##### `param_offload` 和 `optimizer_offload` 的作用

这两个参数控制了 FSDP 在分片的基础上，是否进一步将模型参数和优化器状态从 **GPU 显存** 卸载（offload）到 **CPU 系统内存 (RAM)**。
1.  **`param_offload=True`**:
    *   **作用**: 当设置为 `True` 时，FSDP 会将模型参数（parameters）从 GPU 显存卸载到 CPU 系统内存中。
    *   **优点**: 进一步减少了每个 GPU 的显存占用。对于那些模型参数非常大，即使分片后单个 GPU 显存仍然不足以容纳的情况，这可以帮助模型跑起来。
    *   **缺点**:
        *   **增加系统内存 (RAM) 占用**: 这是你遇到的主要问题。模型参数从 GPU 转移到 CPU，会占用大量的系统内存。
        *   **增加通信开销和训练速度下降**: 当需要使用模型参数进行前向传播或反向传播时，这些参数需要从 CPU 传输回 GPU。CPU 和 GPU 之间的数据传输速度远低于 GPU 内部的传输速度，这会导致训练速度显著下降。
2.  **`optimizer_offload=True`**:
    *   **作用**: 当设置为 `True` 时，FSDP 会将优化器状态（optimizer states，例如 Adam 优化器中的动量和方差估计）从 GPU 显存卸载到 CPU 系统内存中。
    *   **优点**: 优化器状态通常比模型参数本身还要大（例如，Adam 优化器通常需要两倍于模型参数的存储空间）。将其卸载到 CPU 可以显著减少 GPU 显存占用。
    *   **缺点**:
        *   **增加系统内存 (RAM) 占用**: 同样，这会占用大量的系统内存。
        *   **增加通信开销和训练速度下降**: 优化器在每次迭代中都需要更新其状态，如果状态在 CPU 上，就需要频繁地在 CPU 和 GPU 之间传输，导致训练速度变慢。
