**城投公司数据资产：**
与湖南省宁乡市城投公司进行合作。
城投公司面临着数据资产困境，包括数据确权难题以及沉睡数据有大量的未挖掘价值。
1. 数据确权难题
由于城投公司的数据来源十分广泛，设计公共事务、基础设施等诸多领域，确权的流程十分复杂，并且缺乏明确的标准，导致数据资产的归属十分模糊。 -> 数字水印机
2. 沉睡数据价值待挖掘
大量的数据仅仅是简单的记录和统计，没有深度挖掘其潜在价值，比如交通流量的数据并没有用来优化城市规划以及商业布局，数据资产处于沉睡状态，未能转化为经济效益。

在第二个问题的背景下，我们推出了数据资产运营平台这一产品。

产品的使用流程主要分成三个步骤：
1. 加载数据源：提供多种数据源接入方式，支持城投公司内部各类系统数据、外部合作机构数据等快速导入，确保数据完整性与准确性，为后续处理奠定基础。
2. 加载模型算法：模型算法库丰富，涵盖数据分析、预测、挖掘等类型，类似应用商城APP，按需导入加载，满足不同业务场景需求，如城市人口流动预测助力精准规划。
3. 导出计算结果：结果导出格式多样，适配城投公司多种业务系统与决策需求，支持可视化展示，让复杂数据直观呈现，辅助管理层高效决策，推动数据资产价值落地。

#### 2.1.2 消息历史处理
##### LangChain中原生组件使用：
 1. ChatMessageHistory（消息存储）
```python
  from langchain_community.chat_message_histories import ChatMessageHistory
  self._histories: Dict[str, ChatMessageHistory] = {}  # session_id -> history对象
```
  作用：
  - 标准的消息历史容器
  - 自动管理消息列表：history.messages = [HumanMessage(...), AIMessage(...), ...]
  - 提供便捷方法：add_user_message(), add_ai_message()
  结构：
  ```
	history = ChatMessageHistory()
  history.add_user_message("我今天头有点晕")
  history.add_ai_message("头晕可能由...")
  # 内部存储
  history.messages = [
      HumanMessage(content="我今天头有点晕"),
      AIMessage(content="头晕可能由...")
  ]
  ```
  
  2. MessagesPlaceholder（Prompt占位符）
  ```
    prompt = ChatPromptTemplate.from_messages([
      ("system", system_msg),
      MessagesPlaceholder(variable_name="history"),  # ← LangChain标准组件
      ("human", user_msg),
  ])
  ```
  作用：
  - 在Prompt模板中预留历史消息的位置
  - 运行时自动注入历史消息列表
  - 支持动态替换

  3. RunnableWithMessageHistory（自动历史管理）
```
  from langchain_core.runnables import RunnableWithMessageHistory

  self.rag_chain = RunnableWithMessageHistory(
      core_chain,                    # 核心业务链
      _get_history_wrapper,          # 历史获取函数
      input_messages_key="original_input",     # 用户输入的字段名
      history_messages_key="history",          # 历史注入的字段名
      output_messages_key="answer",            # AI输出的字段名
  )
```

  作用：
  - 自动化历史管理：无需手动添加消息到历史
  - 自动注入历史：运行时从存储中获取历史并注入
  - 自动保存输出：执行完成后自动将新消息添加到历史

##### 项目创新记忆机制：
1. 三层记忆架构
```
  prompt = ChatPromptTemplate.from_messages([
      ("system", system_msg),  # ← 包含 running_summary（长期记忆）
      MessagesPlaceholder("history"),  # ← 短期记忆（LangChain标准）
      ("human", user_msg),  # ← 即时上下文
  ])
```
  - 长期记忆：running_summary（压缩的早期对话）
  - 短期记忆：history.messages（最近的完整对话）
  - 即时上下文：当前问题 + 检索文档

2. 历史压缩机制与token估算
先计算平均每个字符的token数量，用此计算下一次对话的消耗，如果超过阈值，则触发历史压缩机制。
压缩机制：
- 短期记忆的前50% → 通过LLM压缩成摘要
- 摘要追加到长期记忆（running_summary += 新摘要
- 短期记忆裁剪（只保留后50%）

#### 2.1.3 检索器集成
langchain提供了标准接口BaseRetriever，使用milvus实现了混合搜索的所有相关逻辑。
### 2.2 Milvus2.6.x,多向量储存，混合检索，数据流水线
#### 2.2.1 多向量储存
- 知识库储存了三个向量字段

| 字段名           | 向量类型 | 维度   | 存储内容      | 实现方式        |     |
| ------------- | ---- | ---- | --------- | ----------- | --- |
| summary_dense | 稠密向量 | 1024 | 问题摘要的向量化  | Embedding模型 |     |
| text_dense    | 稠密向量 | 1024 | 完整文档的向量化  | Embedding模型 |     |
| text_sparse   | 稀疏向量 | 动态   | 文本的BM25权重 | 自管理BM25     |     |
- 档案储存了1个向量字段

| 字段名          | 向量类型 | 维度   | 存储内容     | 实现方式        |
| ------------ | ---- | ---- | -------- | ----------- |
| chunk_vector | 稠密向量 | 1024 | 档案文本的向量化 | Embedding模型 |
1. 稠密向量的实现
- 使用的是阿里的text-embedding-v4
2. 稀疏向量的实现
- 使用了自管理的BM25
#### 2.2.2 混合检索的实现
整个流程中一共涉及到三次检索。
1. 用户的查询到达后，使用稠密向量检索在用户档案库中检索相关的条目，使用条目作为背景信息改写用户查询。
2. 改写完毕之后，进行知识库查询
- 使用问答数据集中的问题和改写后的查询问题做稠密向量检索匹配，返回top10的结果
- 使用问答数据集中的问题 + 回答进行BM25检索匹配，返回top10的结果
- 最后进行加权融合，取前top10的结果
```
  用户查询："高血压患者饮食注意事项"
      ↓
  【档案增强】（如果有elder_id）
  档案检索 → 改写："68岁高血压患者...饮食注意事项"
      ↓
  【历史改写】（仅MultiDialogueRag）
  结合历史 → 改写："68岁高血压合并糖尿病...饮食应该注意什么？"
      ↓
  【向量化】
  summary_dense_vector = embed_query(...)  # [0.234, -0.567, ...]
  text_sparse_vector = bm25_vectorize(...)  # {0: 2.8, 1: 1.5, ...}
      ↓
  【构建子查询】
  AnnSearchRequest(summary_dense_vector, "summary_dense", limit=10)
  AnnSearchRequest(text_sparse_vector, "text_sparse", limit=10)
      ↓
  【Milvus并行检索】
  线程1：summary_dense检索 → Top10结果
  线程2：text_sparse检索 → Top10结果
      ↓
  【分数归一化】
  summary_dense分数归一化到[0, 1]
  text_sparse分数归一化到[0, 1]
      ↓
  【加权融合】
  final_score = 0.8 × normalized_dense + 0.2 × normalized_sparse
      ↓
  【排序】
  按final_score降序排序 → Top10
      ↓
  【结果封装】
  List[Document(page_content, metadata)]
      ↓
  【上下文构建】
  智能文档截断（Token预算管理）
      ↓
  【LLM生成】
  基于检索结果生成个性化答案
```

## 三、召回率与精确率
| 检索方式     | 召回率        | 精确率        | 适用场景     |
| -------- | ---------- | ---------- | -------- |
| 仅稠密向量    | 70.12%     | 85.64%     | 语义相似问题   |
